# inspireFlow / 升变 PAWN 开发交接说明

## 1. 阅读顺序

开始开发前，按以下顺序阅读：

1. `Agent.md`：当前开发约束、项目现状与交接说明，优先级最高。
2. `MVP.md`：当前 iOS MVP 的目标和验收标准。
3. `newIDEA.md`：完整产品背景、用户价值与长期规划。
4. `protocol.md`：戒指 v4 BLE 协议字段。
5. `ring_sound_use.md`：Python 戒指 SDK 的公开调用方式。
6. `README-RingSDK.md`：戒指 SDK 的实现边界、音频格式和设备行为。
7. `AUDIT.md`、`PLAN-TEMPLATE.md`：仅用于动画审查，不属于产品需求。

如果文档之间发生冲突，以本文件和设备当前实际协议为准。

## 2. 产品定位

PAWN 是面向 Bilibili 创作者的无屏 AI 制片助理。

核心链路：

现场灵感 → 戒指表达意图 → 耳机承载语音和私密反馈 → PAWN 维护项目上下文 → 生成 Bilibili 创作材料 → 商业项目进入验收、授权与结算。

职责边界：

- 戒指负责开始、确认、取消、撤销和手势控制。
- viaim 耳机负责主要语音输入和私密语音反馈。
- 戒指自身录音能力可以作为硬件演示或备用输入，但不应模糊产品主线。
- PAWN 负责维护项目上下文、追问和生成结构化创作成果。
- Bilibili 定义目标创作者与最终内容形态。
- Injective 只处理商业项目的预算托管、版本验收、授权和分账。
- 普通灵感、原始录音、聊天、脚本和草稿不得默认上链。

## 3. 当前平台与技术栈

主要客户端是 iOS 应用：

- Swift
- SwiftUI
- Swift Concurrency
- UserDefaults 本地持久化
- Apple 原生框架优先
- 不引入不必要的第三方依赖

戒指现有 SDK 是 Python 实现，不能直接导入 SwiftUI 应用。

iOS 接入戒指时应使用 CoreBluetooth，按照 `protocol.md` 实现 Nordic UART Service 和 v4 数据包协议；也可以在演示阶段通过独立桥接服务连接，但不能把 Python API 描述成 iOS 原生 API。

## 4. 当前信息架构

项目已经存在创作者和品牌方两种角色：

### 创作者

- 工作台
- 创作项目
- 灵感捕捉与详情
- 项目内 PAWN 协作
- 我的

### 品牌方

- 商业工作台
- 商业委托
- 消息
- 我的

不要在没有明确需求时再次新增“活动”“设备”或独立的全局 PAWN Tab。

PAWN 应优先存在于具体项目上下文中，而不是成为与项目脱离的通用聊天机器人。

角色当前是账号会话级切换；未来如需项目级角色，应先扩展项目成员和权限模型，不要只通过界面条件判断模拟权限。

## 7. 当前 MVP 主流程

开发优先保证以下链路始终可运行：

1. 用户以创作者身份进入工作台。
2. 用户新建项目。
3. 项目保存到本地并出现在项目列表。
4. 用户从工作台或项目详情开始捕捉灵感。
5. 应用明确标记当前录音和转写属于演示数据。
6. PAWN 完成三轮追问。
7. 应用生成 Bilibili 创作方案。
8. 灵感和方案保存到本地。
9. 灵感可以关联到项目。
10. 用户可以从项目详情进入项目内 PAWN 对话。
11. 用户可以导出创作方案。

品牌方流程当前只需要支持：

1. 发布商业委托。
2. 在商业委托列表中查看项目。
3. 查看项目阶段。
4. 通过演示操作推进验收与结算状态。

不要为了未来完整产品破坏这两条已经存在的演示链路。

## 8. 视觉设计规范

当前界面采用深色、黑白双核心色设计。

必须遵守：

- 核心颜色只使用黑色、白色以及它们的透明度。
- 页面背景使用近黑色。
- 卡片使用半透明白色、系统 Material 和细微描边。
- 主操作使用白底黑字。
- 次要操作使用透明白底和白色文字。
- 使用系统字体和 SF Symbols。
- 默认水平页面边距使用 `ShengbianMetrics.pageMargin`。
- 默认卡片、控件圆角使用设计令牌。
- 不新增蓝色、紫色、红色、绿色或彩色渐变。
- 状态不能只依赖颜色，必须同时使用文字、图标或形状表达。
- 不在业务 View 中硬编码新的颜色值。
- 优先复用 `ShengbianColors`、`ShengbianTypography`、`ShengbianMetrics`、`ShengbianGlassCard` 和 `ShengbianBackground`。

目前部分状态仍以 `listening`、`success`、`warning` 命名，但它们应映射为黑白明暗层级，而不是彩色强调。

## 9. 动效与无障碍规范

- 按压反馈缩放约为 `0.97`。
- 按压时长为 100 至 160 毫秒。
- 普通状态切换使用 150 至 250 毫秒的 `easeOut`。
- 高频操作避免装饰性动画。
- 不使用持续循环的漂浮、呼吸发光或大幅位移动画。
- 读取 `accessibilityReduceMotion`。
- 减少动态效果开启时取消缩放和位移，保留必要的透明度反馈。
- 支持 Dynamic Type。
- 图标按钮必须有无障碍标签。
- 状态信息不得只通过颜色表达。
- 录制、设备断连和生成完成等关键变化应提供可读状态文字。
- 新交互应在 VoiceOver 下保持合理顺序。

## 10. 戒指协议边界

iOS 端接入戒指时必须注意：

- 使用 Nordic UART Service。
- Service UUID：`6E400001-B5A3-F393-E0A9-E50E24DCCA9E`
- TX UUID：`6E400003-B5A3-F393-E0A9-E50E24DCCA9E`
- RX UUID：`6E400002-B5A3-F393-E0A9-E50E24DCCA9E`
- 协议版本为 v4。
- 协议头固定为 11 字节。
- 普通协议字段使用大端序。
- 录音文件内部 Speex 长度字段使用 2 字节小端序。
- BLE 通知可能包含半包、整包或多个连续包，必须实现流式重组。
- 不能把一次 BLE 通知直接当作完整协议包。
- 当前协议没有可靠的“查询设备模式”命令。
- `0x0704` 只说明设备识别到一次按键单击，不证明录音模式或手势模式切换成功。
- `0x0601` 只开启实时 IMU BLE 上报，不负责启动本地 IMU。
- 只有设备已经进入手势模式后，`start_sensor_report` 对应流程才可能成功。
- `0x0505` 可能来自普通下载、快速下载或录音保存后的主动上报。
- 同一连接上不能让多个流程同时消费同一个录音数据命令队列。
- 删除戒指录音属于破坏性操作，必须二次确认。

协议实现发生冲突时，以设备实际行为、`protocol.md` 和 `ring_sound.py` 当前实现共同核对。

## 11. 数据与隐私边界

当前本地持久化适合 MVP 演示，但不能宣传为已经完成高强度加密。

界面中如果显示“本地内容已加密”，必须确认实际已经实现对应的加密存储；如果只是 UserDefaults，应改成“内容保存在本地”或明确标记为演示状态。

以下数据默认不上链：

- 原始录音
- 实时转写
- 私人对话
- 完整脚本
- 未发布草稿
- 用户个人信息
- 每一次编辑操作

商业项目未来可以上链的数据包括：

- 商业任务标识
- 托管预算状态
- 作品版本摘要
- 授权范围
- 验收状态
- 分账比例
- 结算交易

不要使用“内容哈希证明原创”或“上链确权”等不准确表达。

## 12. 当前已知问题

### 编译问题

`ProjectDetailView.swift` 曾出现中文弯引号：

- `“`
- `”`

Swift 字符串必须使用英文直引号：

- `"`

如果当前编辑器内容已经是英文引号，但编译器仍报错，应依次检查：

1. 保存全部文件。
2. 全项目搜索 `“` 和 `”`。
3. 检查是否存在重复的 `ProjectDetailView.swift`。
4. 检查 Target Membership 是否包含旧副本。
5. 执行 Clean Build Folder。
6. 删除 Derived Data 后重新构建。

`Expected '}'` 很可能是弯引号导致的连锁语法错误，不应在没有确认前随意增加大括号。

### 产品表达问题

- `AccountView` 当前显示“本地内容已加密”，但需要核实数据层是否真的加密。
- 真实语音识别尚未接入，必须继续显示“演示数据”。
- PAWN 回复是本地模板生成，不得描述成真实后端 Agent。
- 项目阶段目前可手动推进，仅适合演示。
- 商业预算当前被拼接进 `initialIdea`，不是正式结构化字段。
- 项目成果入口部分仍是空状态。
- 品牌方与创作者共用基础项目详情，尚未建立真正的项目权限系统。
- 项目命名中同时出现 `inspireFlow`、`升变` 和 `PAWN`，后续需要统一品牌层级。
- 根目录 `README.md` 是第三方设计 Skill 说明，不是产品 README。

## 13. 下一步优先级

按照以下顺序进行小步开发：

1. 保证项目可以无错误编译。
2. 清理弯引号、重复文件和无效 Target Membership。
3. 核实 `AppStore` 的当前模型和持久化迁移。
4. 将“本地内容已加密”改成符合真实实现的文案。
5. 保证新建项目、灵感捕捉、关联项目和项目 PAWN 闭环稳定。
6. 为关键 AppStore 操作补充 Swift Testing 测试。
7. 将商业项目的预算和截止时间从 `initialIdea` 拆为结构化字段。
8. 使用 Speech 或 SpeechAnalyzer 接入真实转写，保留权限拒绝和演示回退。
9. 使用 CoreBluetooth 实现 iOS 戒指连接与协议层。
10. 接入真实 PAWN 后端。
11. 最后接入 Injective 商业托管和结算。

每次只完成一个可验证的闭环，不要同时重构导航、数据模型、蓝牙和 Agent。

## 14. 开发边界

未经明确要求，不要：

- 重写整个导航结构。
- 新增第三方 UI 框架。
- 将 Python SDK 直接复制进 iOS Target。
- 将 UserDefaults 描述为安全加密数据库。
- 将演示生成描述为真实 AI。
- 将灵感、聊天或草稿上链。
- 使用哈希宣称法律意义上的原创证明。
- 给普通个人项目强制增加钱包流程。
- 创建第二套颜色、字体或间距系统。
- 在多个 View 中复制相同玻璃卡片组件。
- 一次提交大范围修改无关文件。

## 15. 修改与交付要求

修改现有文件时：

- 保留无关功能。
- 避免重复定义已有类型。
- 使用项目现有命名与设计令牌。
- 确认 Preview 所需的 EnvironmentObject 已注入。
- 说明修改了哪些文件。
- 说明是否涉及数据迁移。
- 说明如何手动验证。
- 编译失败时先报告第一条真实错误，不要追逐后续连锁错误。
- 不要用富文本编辑器生成 Swift 源码，避免中文弯引号进入代码。

完成任务前至少验证：

- App 可以构建。
- 新建项目可保存。
- 灵感可完成演示生成。
- 灵感可关联到项目。
- 项目详情可打开。
- 项目内 PAWN 可发送、停止和重新生成。
- 重启后本地数据仍存在。
- VoiceOver 标签没有明显缺失。
- 减少动态效果不会出现不必要的缩放和位移。
