# PAWN：面向 Bilibili 创作者的无屏 AI 制片助理

## 一、项目概述

PAWN 是一套面向 Bilibili 创作者的环境式 AI 创作工作流。

它通过智能戒指识别创作者的操作意图，通过耳机捕捉语音并提供私密反馈，由 AI 持续维护创作项目上下文，把现场产生的零散灵感逐步整理成可拍摄、可发布的视频方案。

当项目涉及品牌委托、多人协作或商业授权时，PAWN 使用 Injective 完成预算托管、授权确认和协作者自动分账。

PAWN 连接的不只是几个硬件或平台，而是创作者从产生灵感到获得收入的完整路径。

---

## 二、一句话定位

> PAWN 是面向 Bilibili 创作者的无屏 AI 制片助理，从现场灵感一直推进到视频发布、商业交付与可信结算。

### 英文定位

> PAWN is an ambient AI production assistant for Bilibili creators, turning ideas captured through wearables into publishable content and trusted commercial delivery.

### 三段式价值主张

- Capture without a screen.
- Create with context.
- Get paid with certainty.

中文：

- 无屏捕捉；
- 带上下文创作；
- 确定性结算。

---

## 三、要解决的问题

创作者并不总是在桌前产生灵感。

真正有价值的选题、表达和现场观察，经常出现在以下场景：

- 行走途中；
- 活动现场；
- 拍摄过程中；
- 与人交谈时；
- 双手正在操作设备时；
- 不适合拿出手机时；
- 无法立即整理内容时。

传统工具要求创作者停下当前行为、打开屏幕、找到应用并手动记录。这不仅会打断现场状态，还容易导致语境、情绪和后续行动丢失。

因此，PAWN 要解决的不是简单的“语音记笔记”，而是：

> 从现场灵感到可执行视频项目之间的断层。

---

## 四、统一产品主线

所有赛道和技术能力服务于同一条创作生命周期：

1. 创作者在现场产生灵感；
2. 使用智能戒指表达“开始捕捉”的意图；
3. 耳机捕捉创作者的表达；
4. PAWN 判断上下文是否充分，并通过耳机提出简短追问；
5. 创作者通过语音回答，并用戒指完成确认、撤销或隐私设置；
6. PAWN 把内容关联到已有项目，或创建新的创作项目；
7. PAWN 生成视频选题、大纲、分镜、拍摄清单和提词稿；
8. 创作者完成视频并发布到 Bilibili；
9. 如果项目属于商业委托，则通过 Injective 完成验收、授权、付款和协作者分账。

完整流程：

> 现场灵感 → 戒指表达意图 → 耳机承载内容 → PAWN 维护上下文 → Bilibili 视频创作 → Injective 商业结算

## 五、各赛道的明确职责

### 5.1 Bilibili：定义创作者和最终产物

Bilibili 不是一个附加的“发布按钮”，而是产品服务对象和创作输出的核心。

PAWN 主要服务：

- 活动现场 Vlogger；
- 科技区创作者；
- 知识区创作者；
- 采访型 UP 主；
- 多人视频制作团队；
- 接受品牌委托的创作者；
- 需要频繁捕捉现场素材的内容团队。

PAWN 的输出应直接面向视频创作，包括：

- 选题卡；
- 标题候选；
- 三秒开场钩子；
- 视频大纲；
- 分镜方案；
- 拍摄清单；
- 采访问题；
- 提词稿；
- Bilibili 视频简介；
- 章节时间轴；
- 封面文案；
- 评论区互动问题；
- 商业合作交付清单。

Bilibili 赛道的核心价值是：

> 帮助创作者把现场出现的想法，快速推进为真正可以制作和发布的视频项目。

---

### 5.2 智能戒指：意图识别与无屏控制

智能戒指的主要价值不是替代麦克风，而是在创作者无法看屏幕时表达操作意图。

职责定义：

> 戒指负责“现在要做什么”。

建议的手势映射：

| 戒指操作 | PAWN 行为              |
| -------- | ---------------------- |
| 普通双击 | 开始捕捉当前灵感       |
| 长按     | 开始一段连续表达       |
| 松开     | 结束表达并让 PAWN 整理 |
| 向前旋转 | 确认操作或加入当前项目 |
| 向后旋转 | 返回、拒绝或标记为私密 |
| 挥动     | 忽略或撤销刚才的内容   |
| 按键单击 | 切换录音模式与手势模式 |
| 按键双击 | 执行用户预设的快捷操作 |

手势应具有明确的业务语义，例如：

- 这是一个新灵感；
- 把它加入当前项目；
- 这段内容保持私密；
- 接受 PAWN 的建议；
- 撤销刚才的操作；
- 完成当前表达。

设计原则：

- 高频操作必须简单；
- 手势数量保持克制；
- 每次操作都应提供明确反馈；
- 不使用手势完成复杂文本编辑；
- 不要求用户记忆大量组合手势；
- 付款、授权等关键商业操作不能只通过手势确认。

---

### 5.3 耳机：语音输入与私密反馈

耳机负责捕捉创作者的表达，并让 PAWN 在不打断现场环境的情况下提供反馈。

职责定义：

> 耳机负责“创作者说了什么，以及 PAWN 如何回应”。

核心能力：

- 捕捉创作者的连续表达；
- 记录现场采访和环境声音；
- 播放 PAWN 的简短确认；
- 通过私密语音提出追问；
- 在用户不看屏幕时反馈操作结果；
- 提醒创作者缺失的拍摄素材。

典型交互流程：

1. 用户双击戒指；
2. 耳机开始捕捉表达；
3. 用户说出当前灵感；
4. PAWN 判断信息是否充分；
5. PAWN 通过耳机提出一句简短追问；
6. 用户直接回答；
7. 用户用戒指确认项目或隐私状态；
8. PAWN 在后台创建或更新创作项目。

PAWN 可以提出的问题包括：

- “这是口播还是现场采访？”
- “目标观众是谁？”
- “要加入当前项目吗？”
- “需要保留现场声音吗？”
- “这段内容是否保持私密？”
- “要把它整理成一分钟视频吗？”
- “是否生成对应的拍摄清单？”

如果耳机只承担录音功能，手机麦克风也能完成。耳机真正的价值在于：

- 不需要持续看屏幕；
- 反馈只有创作者本人能听见；
- 可以在行走和工作时继续思考；
- PAWN 的追问不会向周围广播；
- 与戒指共同构成完整的无屏交互闭环。

因此二者的职责可以总结为：

> 戒指负责意图和控制，耳机负责内容和反馈。

---

### 5.4 PAWN：持续维护上下文的创作 Agent

PAWN 不是一次性的聊天机器人，而是持续维护项目状态、创作上下文和下一步行动的 AI 制片助理。

名称可以解释为：

> Personal Ambient Workflow Navigator

含义：

- Personal：理解个人创作习惯；
- Ambient：通过戒指和耳机自然工作；
- Workflow：维护完整创作流程；
- Navigator：引导灵感逐步成为作品。

这只是团队对 PAWN 的产品定义，不应宣称它一定是官方提示的唯一答案。

PAWN 的主要职责：

- 判断输入属于哪个项目；
- 补充缺失的创作上下文；
- 把语音整理成结构化灵感；
- 根据内容创建视频选题；
- 生成脚本、大纲和分镜；
- 生成现场拍摄清单；
- 保持多轮讨论的一致性；
- 记录项目当前阶段；
- 提醒待完成事项；
- 为 Bilibili 生成发布材料；
- 在商业项目中整理交付、授权和结算信息。

品牌隐喻：

国际象棋中的 pawn 看似微小，却能通过持续前进完成升变。

> Every great work starts as a small idea. PAWN helps it move forward, one step at a time, until it becomes something more.

中文表达：

> 每个作品最初都只是一枚微小的灵感。PAWN 让它持续向前，最终完成升变。

这个隐喻适合作为品牌故事，但不需要把整个界面设计成棋盘或棋类游戏。

---

### 5.5 Injective：商业合作的可信执行层

Injective 不参与每一次灵感捕捉，也不应该要求所有普通创作者连接钱包。

它只在以下情况出现：

- 品牌方委托创作者制作视频；
- 活动方发布内容创作任务；
- 多位创作者共同制作作品；
- 摄影、出镜和剪辑需要按比例分账；
- 品牌购买视频或素材的商业使用权；
- 创作者需要确认客户已经准备好预算。

职责定义：

> Injective 负责作品进入商业合作后的预算托管、授权确认和自动结算。

Injective 解决的问题：

- 创作者如何确认客户确实准备了预算；
- 客户如何确认付款对应某个具体作品版本；
- 商业授权在什么时间生效；
- 多位协作者如何按照约定自动分账；
- 如何跨平台验证合作和结算记录；
- 如何避免中心化平台单方面修改合作状态。

Injective 不负责：

- 保存原始录音；
- 保存私人对话；
- 保存完整脚本；
- 保存未发布草稿；
- 记录每一次编辑；
- 让每条灵感上链；
- 把每个作品铸造成 NFT；
- 宣称内容哈希能够证明原创。

适合上链的关键事件：

- 商业任务预算托管；
- 创作者接受明确任务；
- 协作者确认收益比例；
- 提交可验收的作品版本；
- 委托方确认验收；
- 授权条款生效；
- 资金释放；
- 协作者自动分账。

建议记录的数据：

- 项目标识；
- 作品版本哈希；
- 委托方地址；
- 创作者地址；
- 授权类型；
- 授权期限；
- 预算金额；
- 收益分配比例；
- 项目状态；
- 交易时间；
- Bilibili 交付链接的摘要或引用。

原始内容和个人信息仍然保存在本地或加密存储中。

## 六、为什么区块链不再牵强

如果区块链只用于保存灵感哈希，评委很容易质疑：

- 为什么不用普通数据库？
- 谁需要验证这条记录？
- 哈希能否证明原创？
- 为什么一定要使用 Injective？

因此，区块链不能只是“证明层”，必须参与业务执行。

PAWN 中更自然的链上价值是：

1. 客户在创作开始前托管预算；
2. 创作者可以确认资金已经锁定；
3. 创作者提交明确的作品版本；
4. 客户验收后，付款和授权同时生效；
5. 合约自动把收入分配给多位协作者；
6. 任何人可以查看公开的授权和结算凭证。

核心状态关系：

> 验收成功 = 授权生效 = 资金释放 = 协作者完成分账

这让 Injective 不再是附加功能，而是商业委托项目的可信执行基础设施。

## 七、两种项目模式

为了避免 Web3 干扰普通创作者体验，PAWN 应明确区分个人项目和商业委托项目。

### 7.1 普通个人创作

流程：

> 捕捉 → 整理 → 脚本 → 拍摄 → 发布

特点：

- 不需要连接钱包；
- 不需要使用区块链；
- 不需要支付交易；
- 内容保存在本地或私密云端；
- 可以直接生成 Bilibili 发布材料；
- Injective 相关信息不应出现在普通工作流中。

### 7.2 商业委托项目

流程：

> 接受任务 → 验证托管预算 → 捕捉与创作 → 邀请协作者 → 确认收益比例 → 提交作品 → 委托方验收 → 授权生效 → 付款与分账

只有商业委托项目才需要 Injective。

界面应优先使用普通业务语言：

- 预算已托管；
- 等待验收；
- 授权已生效；
- 收益已分配；
- 查看公开凭证。

钱包地址、合约地址和交易哈希应放在详情页，不应成为产品首页的视觉中心。

## 八、核心商业场景：活动现场创作任务

这是最容易把 Bilibili、智能戒指、耳机、PAWN 和 Injective 连接起来的业务场景。

示例任务：

| 字段     | 内容                                 |
| -------- | ------------------------------------ |
| 主题     | 用 60 秒介绍黑客松现场的三个创新项目 |
| 预算     | 500 USDT                             |
| 交付平台 | Bilibili                             |
| 内容形式 | 竖屏短视频                           |
| 截止时间 | 今天 18:00                           |
| 授权方式 | 委托方商业使用                       |
| 结算方式 | Injective 托管与自动分账             |

完整业务流程：

1. 主办方发布现场创作任务；
2. 预算进入 Injective 托管；
3. Bilibili 创作者接受任务；
4. 创作者通过戒指和耳机捕捉现场素材；
5. PAWN 生成采访问题、视频结构和拍摄清单；
6. 创作者邀请摄影师共同参与；
7. 双方确认收益分配比例；
8. 创作者完成并发布视频；
9. 提交 Bilibili 链接和作品版本；
10. 主办方确认验收；
11. 商业使用授权生效；
12. Injective 自动释放资金并完成分账。

各项能力在这个场景中的职责：

| 能力      | 职责                             |
| --------- | -------------------------------- |
| Bilibili  | 定义创作者、视频形态和发布目标   |
| 智能戒指  | 无屏触发、确认、撤销和隐私控制   |
| 耳机      | 捕捉表达、播放反馈并提出私密追问 |
| PAWN      | 维护上下文并推进创作项目         |
| Injective | 托管预算、确认授权并自动分账     |

## 九、黑客松 Demo 脚本

展示时不要逐个介绍五个赛道功能，而应该讲述一个创作者完成商业任务的故事。

### 第一幕：接受任务

应用显示：

| 项目字段 | 内容                   |
| -------- | ---------------------- |
| 创作主题 | 用 60 秒介绍黑客松现场 |
| 预算     | 500 USDT               |
| 预算状态 | 已托管                 |
| 交付平台 | Bilibili               |
| 视频形式 | 竖屏短视频             |
| 截止时间 | 今天 18:00             |

用户点击“接受任务”。

这一幕证明 Injective 的直接价值：

> 创作者开始工作前，可以确认客户的预算已经真实锁定。

### 第二幕：现场无屏捕捉

演示者双击戒指，对耳机说：

> 这个项目把硬件手势变成无屏创作控制。我想用创作者在现场忘记灵感作为视频开头。

PAWN 通过耳机回应：

> 已记录。要加入“黑客松现场 60 秒”项目吗？

用户通过旋转戒指确认。

应用显示：

> 已加入项目：黑客松现场 60 秒

这一幕同时展示：

- 戒指识别操作意图；
- 耳机捕捉创作内容；
- PAWN 提出上下文追问；
- 用户通过手势完成无屏确认；
- 输入被正确关联到持续存在的项目。

### 第三幕：PAWN 生成创作方案

PAWN 根据现场内容自动生成：

- 三秒开场钩子；
- 三个采访对象建议；
- 三段项目介绍；
- 现场拍摄清单；
- 六十秒视频脚本；
- Bilibili 标题候选；
- 视频简介；
- 评论区互动问题。

这一幕的核心不是展示“AI 会写文案”，而是：

> PAWN 知道刚才捕捉的内容属于哪个任务，并持续维护完整项目上下文。

### 第四幕：设置协作者分账

创作者邀请摄影师参与：

| 角色     | 收益比例 |
| -------- | -------: |
| 主创作者 |      70% |
| 摄影师   |      30% |

双方确认后，分配方案成为任务结算条件。

### 第五幕：发布与结算

视频发布到 Bilibili 后，创作者提交链接。委托方确认验收，应用显示：

| 结算字段     | 结果           |
| ------------ | -------------- |
| 任务状态     | 已完成         |
| 支付金额     | 500 USDT       |
| 主创作者收入 | 350 USDT       |
| 摄影师收入   | 150 USDT       |
| 授权状态     | 商业使用已生效 |
| 结算网络     | Injective      |

用户可以点击“查看公开凭证”打开真实链上交易记录。

这构成了从灵感产生到获得收入的完整闭环。

## 十、产品信息架构

建议使用四个主要入口：

> 捕捉 / 项目 / PAWN / 我的

### 10.1 捕捉

显示：

- 智能戒指连接状态；
- 耳机连接状态；
- 当前绑定项目；
- 当前捕捉状态；
- 当前内容可见性；
- 最近捕捉的灵感；
- 可用手势提示；
- PAWN 是否正在聆听。

### 10.2 项目

显示：

- 个人创作项目；
- 商业委托项目；
- 当前创作进度；
- 项目截止时间；
- 托管预算状态；
- 待提交里程碑；
- 待验收内容；
- 已完成项目。

### 10.3 PAWN

承担项目上下文对话：

- 补充创作方向；
- 回答 PAWN 的追问；
- 调整视频结构；
- 生成拍摄清单；
- 完善脚本和分镜；
- 创建提交版本；
- 整理 Bilibili 发布内容；
- 检查商业项目交付要求。

### 10.4 我的

显示：

- 智能戒指；
- 耳机；
- 本地加密；
- Bilibili 账号；
- 商业授权记录；
- 收益和结算；
- 钱包设置；
- 链上凭证。

钱包是商业功能的账户基础设施，不是整个产品的核心入口。

## 十一、现有产品概念调整

### 11.1 拆分“隐私”和“授权”

“私密、授权、公开”不属于同一个维度，不应放在同一个选择器中。

建议拆分为两个概念。

#### 内容可见性

- 仅自己；
- 项目成员；
- 公开。

#### 商业权利

只在作品提交或发布阶段设置：

- 不提供商业授权；
- 非独家商业授权；
- 独家商业授权；
- 委托方使用授权。

### 11.2 移除全局“链上授权证明”开关

授权不是全局设置，而是针对：

- 某个作品；
- 某个具体版本；
- 某个被授权方；
- 某段授权时间；
- 某个使用范围；
- 某个支付条件。

因此应改成项目中的具体操作：

- 查看托管预算；
- 设置收益分配；
- 创建提交版本；
- 确认授权条款；
- 提交验收；
- 查看结算凭证。

### 11.3 不使用“上链确权”作为核心宣传

内容哈希只能证明某个地址在某个时间提交过某个值，不能独立证明：

- 上传者是法律意义上的作者；
- 内容不是抄袭；
- 第一个上链的人就是最早创作者；
- 上传者拥有完整版权。

更准确的产品表达是：

- 可验证版本记录；
- 创作时间证明；
- 授权凭证；
- 贡献协议；
- 结算记录。

## 十二、视觉设计方向

PAWN 的 UI 应体现“创作者工具、实验性硬件、环境式 Agent”，而不是传统 Web3 应用或高饱和营销页面。

### 建议风格

- 深色背景；
- 低饱和冷灰色体系；
- 克制的青灰或蓝紫强调色；
- 清晰的信息层级；
- 少量玻璃材质；
- 避免大面积霓虹发光；
- 避免高饱和粉色和紫色渐变；
- 避免把钱包、代币和链图标放在视觉中心。

### 推荐配色

| 用途       | 建议颜色    |
| ---------- | ----------- |
| 页面背景   | `#07090D` |
| 卡片背景   | `#10141B` |
| 次级容器   | `#161B23` |
| 普通边框   | `#252B36` |
| 激活边框   | `#45515E` |
| 主文字     | `#E7EAF0` |
| 次文字     | `#8B93A1` |
| 主强调色   | `#6FA8A3` |
| 辅助强调色 | `#777DA7` |
| 成功状态   | `#739987` |
| 警告状态   | `#A38B68` |

色彩原则：

> 90% 中性色 + 10% 低饱和强调色。

### 动效原则

引导页属于首次使用场景，可以有适度动效，但所有动效必须有明确目的。

保留：

- 系统分页手势；
- 卡片轻微缩放与淡化；
- 按钮按压反馈；
- 页面状态的短暂交叉淡化；
- 最终进入主界面的简短过渡。

避免：

- 持续漂浮；
- 持续呼吸发光；
- 大幅视差；
- 复杂 3D 翻转；
- 打字机动画；
- 长时间页面转场；
- 纯装饰背景动画。

开启“减少动态效果”时，应取消位移和缩放，仅保留简短淡入淡出。

## 十五、赛道策略

推荐优先级：

1. 主赛道：Bilibili 创作者服务；
2. 核心交互创新：智能戒指与耳机组成的无屏工作流；
3. 核心 Agent：PAWN；
4. 商业闭环：Injective。

展示时不要说：

> 我们为了参加多个赛道，所以加入了这些功能。

应该说：

> 我们从创作者在现场容易丢失灵感的问题出发，发现屏幕不是合适的交互入口。因此使用戒指表达意图，通过耳机承载语言，并让 PAWN 持续维护项目上下文。作品进入商业合作后，再由 Injective 处理预算、授权和结算。

技术能力应当从用户问题中自然推导出来，而不是以赛道数量作为产品设计依据。

## 十六、评委可能提出的问题

### 为什么不用手机？

手机要求创作者停下当前动作、打开屏幕并主动整理。戒指和耳机让创作者在行走、拍摄和交流过程中完成捕捉，并由 PAWN 自动关联项目上下文。

### 为什么需要戒指？

戒指不只是一个录音设备，而是无屏交互中的意图控制器。它负责开始、确认、撤销和隐私操作，让用户无需说出所有控制指令。

### 为什么需要耳机？

耳机不仅负责录音，还提供只有创作者本人能听见的反馈和追问，使 PAWN 可以在公共环境中与用户完成私密交互。

### PAWN 和普通聊天机器人有什么区别？

普通聊天机器人通常只处理当前对话。PAWN 持续维护项目、创作阶段、历史素材、待完成任务和商业交付条件，并主动把新输入关联到对应项目。

### 为什么使用 Injective？

Injective 不用于保存普通灵感，而用于商业委托中的预算托管、版本验收、授权生效和多人自动分账。它参与业务执行，而不仅保存一条哈希记录。

### 为什么不用普通数据库和支付平台？

数据库可以记录状态，但参与者必须信任平台不会修改记录或冻结资金。链上托管允许创作者在开始工作前验证预算已经锁定，并让验收、授权和分账按照公开规则执行。

### 区块链会不会破坏隐私？

原始语音、脚本、对话和草稿不上链。链上只记录商业合作所需的版本摘要、授权状态和结算信息。

### 内容哈希能证明原创吗？

不能。哈希只能提供可验证版本和时间记录，不能独立证明法律意义上的原创或版权归属。因此产品使用“版本记录”和“授权凭证”，而不是夸大为“上链确权”。

## 十七、最终 Pitch

### 30 秒版本

> 创作者最好的灵感经常出现在无法打开屏幕的时候。PAWN 使用智能戒指识别操作意图，通过耳机捕捉表达并提供私密追问，把现场灵感持续整理成面向 Bilibili 的脚本、分镜和拍摄清单。当作品进入商业合作后，Injective 负责托管预算、确认授权并自动向协作者分账。PAWN 让创作者能够无屏捕捉、带着上下文创作，并确定地获得收入。

### 15 秒版本

> PAWN 是面向 Bilibili 创作者的无屏 AI 制片助理：戒指负责意图，耳机负责表达，AI 负责把灵感推进成作品，Injective 负责商业交付后的授权与结算。

### 一句话版本

> PAWN 把创作者从现场灵感到商业收入的完整路径连接起来，而且在最需要捕捉灵感的时候，不要求他们看屏幕。

## 十八、最终结论

PAWN 不应该被定位为 Web3 灵感笔记，也不应该成为多个硬件和平台能力的简单集合。

各项能力的最终关系是：

- 戒指决定何时捕捉以及执行什么操作；
- 耳机承载创作内容和私密反馈；
- PAWN 维护上下文并推动作品完成；
- Bilibili 定义内容形态和目标创作者；
- Injective 处理商业合作中的托管、授权和分账。

产品真正的创新点不是同时使用多项技术，而是：

> 把创作者从产生灵感到获得收入的整个路径连接起来，并且在最需要捕捉灵感的时候，不要求他们看屏幕。
